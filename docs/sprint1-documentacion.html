1. Resumen Ejecutivo del Sprint 1

El Sprint 1 tuvo como objetivo construir la Landing Page pública de StudyMatch: la primera pantalla que ve cualquier visitante del sitio. El foco estuvo en tres pilares: estética de alta calidad, arquitectura de código limpia y escalable, y experiencia de usuario fluida en todos los dispositivos.

🎯
Objetivo

Construir la Landing Page completa, funcional y visualmente premium de la plataforma StudyMatch UNSTA.

✅
Estado Final

Landing Page completa con 6 secciones, animaciones, responsive design y navegación por anclas funcional.

🏗️
Enfoque

CSS-First: estilos en archivos dedicados. Prohibido el uso de @apply de Tailwind y emojis en el código.

Se desarrollaron 6 secciones completas de la Landing Page, cada una como un componente de React independiente con su propio archivo de estilos CSS:

2. Stack Tecnológico Principal

El stack fue elegido para maximizar la velocidad de desarrollo, la calidad del resultado visual y la escalabilidad del proyecto hacia las siguientes páginas de la aplicación.

⚛️
React 19
v19.2.4

¿Por qué? Es la librería de UI más adoptada del mercado. Permite construir interfaces como una colección de componentes reutilizables. Su sistema de estados (useState), efectos (useEffect) y Context API son ideales para una SPA como StudyMatch.

⚡
Vite 8
v8.0.4

¿Por qué? Servidor de desarrollo ultra-rápido con Hot Module Replacement (HMR). Los cambios en el código se reflejan en el navegador en millisegundos sin recargar toda la página. Reemplaza a CRA (Create React App) que está deprecado.

🎨
Vanilla CSS
CSS3 + Custom Props

¿Por qué? Máximo control y flexibilidad. Se eligió un enfoque CSS-First sin dependencia de utilidades de Tailwind (@apply), con variables CSS globales para el sistema de diseño y archivos .css por componente.

🔀
React Router DOM 7
v7.14.0

¿Por qué? Sistema de enrutado del lado del cliente (SPA). Permite navegar entre /, /login, /register y /dashboard sin recargar la página completa, manteniendo el estado de la app.

🎬
Framer Motion 12
v12.38.0

¿Por qué? La librería de animaciones más potente del ecosistema React. Permite animar elementos al entrar en el viewport (whileInView), al hacer hover (whileHover) y manejar animaciones de salida complejas con AnimatePresence.

🔷
Lucide React
v1.3.0

¿Por qué? Librería de íconos SVG diseñada específicamente para React. Reemplaza el uso de emojis en la interfaz con íconos limpios, escalables y estilizables. Cada ícono es un componente que acepta props como size y strokeWidth.

5. Sistema de Diseño — CSS-First

Una de las decisiones más importantes del Sprint 1 fue adoptar un enfoque CSS-First estricto: cada componente tiene su propio archivo de estilos, sin mezclar lógica con estilos y sin usar directivas de Tailwind (@apply).

Variables Globales (index.css)

Todas las variables del sistema de diseño se definen en :root para ser accesibles desde cualquier archivo CSS:

/* Paleta de colores principal */ :root { --color-primary: #1a56db; /* Azul institucional */ --color-primary-hover: #1e429f; /* Azul más oscuro para hovers */ --color-accent: #3b82f6; /* Azul claro para acentos */ --color-secondary: #e2e8f0; /* Gris borde */ --background: #ffffff; /* Fondo principal blanco */ /* Tipografías */ --font-heading: 'Outfit', sans-serif; /* Títulos y logos */ --font-sans: 'Plus Jakarta Sans', sans-serif; /* Cuerpo de texto */ } /* Smooth scroll global para todas las anclas */ html { scroll-behavior: smooth; } /* Offset para que el header fijo no tape secciones */ [id] { scroll-margin-top: 6rem; }

Convención de Nombres de Clases (BEM-like)

Las clases CSS siguen un patrón legible: [componente]-[elemento]-[modificador]

/* Ejemplos del HeaderLanding */ .header-wrapper { } /* El elemento raíz del componente */ .header-wrapper.scrolled { } /* Modificador: estado con scroll */ .header-container { } /* Contenedor interno */ .header-logo-group { } /* Grupo del logo */ .header-logo-title { } /* Texto del logo */ .header-nav-link { } /* Link de navegación */ .header-mobile-menu { } /* Menú móvil */

Técnicas Visuales Implementadas

🪟 Glassmorphism

El header usa backdrop-filter: blur() para crear un efecto de cristal translúcido. El color de fondo tiene transparencia (rgba) para que se vea el contenido de atrás.

🔲 CSS Grid Background

La cuadrícula decorativa de FeaturesSection y HowItWorksSection se crea con background-image de dos linear-gradient ortogonales, sin imágenes externas.

✨ Gradient Text

El texto resaltado del Hero usa background-clip: text + -webkit-text-fill-color: transparent para crear texto con degradado sin SVG ni Canvas.

📐 Position Absolute Center

Los links de navegación del header se centran con position: absolute + left: 50% + transform: translateX(-50%) para un centrado perfecto independiente del contenido lateral.

6. Componentes Desarrollados — Análisis Técnico

HeaderLanding — Profundidad Técnica

El Header es el componente más complejo del Sprint 1 por la cantidad de estados y comportamientos que maneja:

💡

¿Por qué useEffect para detectar scroll?

React no detecta eventos del navegador automáticamente. Necesitamos useEffect para registrar un EventListener en el objeto window cuando el componente se monta, y limpiarlo (cleanup) cuando se desmonta, evitando fugas de memoria.

// Patrón: Scroll detection con cleanup useEffect(() => { const handleScroll = () => { setIsScrolled(window.scrollY > 20); // 20px umbral }; window.addEventListener('scroll', handleScroll); // Registrar return () => window.removeEventListener('scroll', handleScroll); // Cleanup }, []); // [] = solo al montar el componente
⚠️

¿Por qué el delay de 100ms en los links del menú mobile?

Sin el delay, React cierra el menú (re-render) antes de que el navegador procese el click en el ancla (#features). El re-render cancela el evento del anchor. El delay de 100ms le da tiempo al navegador para iniciar el smooth scroll antes de que el componente se re-renderice.

✅

¿Por qué scroll-margin-top: 6rem en [id]?

El header es position: fixed, lo que significa que "flota" sobre el contenido. Sin scroll-margin-top, cuando el browser hace scroll a #features, la sección empieza exactamente en el top de la pantalla, tapada por el header. Con 6rem de margen, el scroll se detiene 6rem antes de que el elemento llegue al top, dejando espacio suficiente.

Framer Motion — Patrones Usados

Prop / API ¿Qué hace? Usado en

8. Principios y Decisiones de Diseño

Cada decisión de arquitectura y diseño fue tomada con un objetivo claro. Acá se documenta el "¿por qué?" detrás de las elecciones más importantes.

🚫 Sin emojis en el código

Los emojis dependen del sistema operativo para su renderizado y pueden verse diferente en Windows, Mac, Android e iOS. Los íconos SVG de Lucide son consistentes en todas las plataformas y totalmente estilizables con CSS.

📁 Un archivo CSS por componente

Facilita encontrar y modificar estilos sin afectar a otros componentes. Cada archivo CSS vive junto a su componente lógicamente. Evita el fenómeno "cascade hell" de CSS global.

🧩 Principio de Responsabilidad Única

Cada componente hace una cosa bien. Por eso CTASection y Footer son componentes separados aunque visualmente están juntos. Esto permite reutilizarlos individualmente en otras páginas.

📱 Mobile-First Responsive

Los estilos base están definidos para mobile. Las media queries (min-width) agregan estilos para pantallas más grandes. Esto garantiza que la versión móvil siempre esté pensada, no sea un "afterthought".

⚡ whileInView sobre animate

Las animaciones de secciones internas usan whileInView (se activan al hacer scroll) en lugar de animate (se activan al cargar). Esto mejora la percepción de performance y el engagement del usuario.

🔗 Anchor Links vs React Router

Los links a secciones de la misma página usan <a href="#id"> (nativo del browser) en lugar de <Link to="#id"> de React Router. Esto asegura compatibilidad con el scroll-behavior: smooth del CSS global.

11. Arquitectura Interna: Context, Hooks, Services y Utils

Estas cuatro carpetas forman el núcleo lógico de la aplicación. Mientras que los componentes se encargan de mostrar la UI, estas carpetas se encargan de cómo funciona la app por dentro: cómo se guarda el usuario logueado, cómo se habla con el servidor y cómo se validan los formularios. Entender estas carpetas es entender la arquitectura real del proyecto.

💡

¿Por qué separar en carpetas distintas?

Principio de Responsabilidad Única (SRP): cada carpeta tiene UNA sola responsabilidad. Si mañana cambia la forma de autenticar (de JWT a OAuth), solo se cambia authService.js, sin tocar los componentes ni la UI.

↓ Valida con Zod (validators.js en /utils/)

📁 /context/ — El Estado Global de la App

La carpeta /context/ implementa el patrón React Context API. Su función es crear un "estado compartido" que cualquier componente puede leer sin necesidad de recibir props.

⚠️

El problema que resuelve: Prop Drilling

Sin Context, para pasar el usuario logueado desde App.jsx hasta un botón dentro de un header dentro de un layout, habría que pasarlo como prop en CADA nivel intermedio. Con Context, cualquier componente lo lee directamente con useAuth().

AuthContext.jsx context

Es el archivo más pequeño pero más importante. Crea el "canal" de datos usando createContext() de React.

  • Solo tiene 3 líneas de código pero es la base de toda la autenticación
  • createContext() devuelve un objeto con un .Provider para dar datos y una referencia para leerlos
  • Se exporta con nombre (export const) porque tanto AuthProvider como useAuth lo necesitan importar
  • El valor por defecto es undefined — si alguien usa useAuth() fuera del Provider, obtendrá un error claro
AuthProvider.jsx context

Es el componente que envuelve toda la app y pone los datos disponibles. Contiene toda la lógica de autenticación del frontend.

  • Estado inicial inteligente: lee localStorage al montar por primera vez, así el usuario no pierde su sesión al recargar la página (F5)
  • función login(): llama a authService → guarda en useState → guarda en localStorage → retorna los datos
  • función register(): llama a authService → retorna la respuesta (no loguea automáticamente)
  • función logout(): limpia useState → borra localStorage → todos los componentes se actualizan
  • Expone todo a través del value del Provider: { user, login, register, logout, loading }
  • loading: preparado para mostrar spinners mientras el backend responde
// ¿Cómo se conecta todo? En main.jsx: createRoot(document.getElementById('root')).render( <React.StrictMode> <AuthProvider> // ← AuthProvider envuelve TODA la app <BrowserRouter> // ← El router va dentro del AuthProvider <App /> // ← Todos los componentes tienen acceso a useAuth() </BrowserRouter> </AuthProvider> </React.StrictMode> )

📁 /hooks/ — Lógica Reutilizable

La carpeta /hooks/ contiene funciones que empiezan con use y pueden usar otros hooks de React. Son el mecanismo para compartir lógica entre componentes sin duplicar código.

useAuth.js custom hook

Un hook de 4 líneas que encapsula el acceso al AuthContext. Es el puente entre el contexto y los componentes.

  • Usa useContext(AuthContext) internamente para leer el contexto
  • Cualquier componente que lo llame recibe: { user, login, register, logout, loading }
  • Si el Provider actualiza sus datos (ej: usuario hace login), el componente que usa useAuth() se re-renderiza automáticamente
  • Oculta el detalle de implementación: los componentes no necesitan saber que existe useContext ni AuthContext
// Uso en cualquier componente — así de simple: function HeaderLanding() { const { user, logout } = useAuth(); // ← Una sola línea return ( <header> {user ? ( <button onClick={logout}>Cerrar sesión, {user.name}</button> ) : ( <Link to="/login">Ingresar</Link> )} </header> ); }

🔁 ¿Qué pasa cuando el usuario hace login?

1. LoginPage llama login() → 2. AuthProvider hace setUser(data) → 3. React detecta cambio de estado → 4. Todos los componentes que usen useAuth() se re-renderizan → 5. El header muestra "Hola Lauty" instantáneamente

💾 ¿Por qué localStorage?

Sin localStorage, al recargar la página (F5), React reinicia su estado a null y el usuario queda deslogueado aunque haya iniciado sesión hace segundos. localStorage persiste los datos entre recargas del navegador.

🔄 ¿Qué es async/await?

Las funciones login() y register() son async porque deben ESPERAR la respuesta del backend (puede tardar 1-3 segundos). await pausa la ejecución hasta que la promesa del servicio se resuelve.

🧩 ¿Por qué useAuth y no useContext directamente?

Si mañana cambia el contexto (renombrarlo, agregar un middleware), solo se cambia el hook. Los 20 componentes que usen useAuth() quedan intactos. Es el principio DRY: Don't Repeat Yourself.

📁 /services/ — Comunicación con el Backend

La carpeta /services/ contiene las funciones que hacen peticiones HTTP al servidor. Es la capa de "acceso a datos" del frontend.

authService.js service

Actualmente es un mock (simulación) que usa Promise + setTimeout en lugar de llamadas HTTP reales. Esto permite desarrollar el frontend independientemente del backend.

  • loginUser(credentials): simula POST /auth/login — responde con usuario + token JWT falso después de 1.5s
  • registerUser(userData): simula POST /auth/register — responde con mensaje de confirmación después de 1.5s
  • Las funciones son async y retornan Promise, exactamente igual a cómo funcionarán con axios
  • Cuando el backend esté listo: solo se reemplaza el cuerpo de las funciones con axios.post() — el resto del código no cambia
// AHORA (mock con setTimeout): export const loginUser = async (data) => { return new Promise((resolve) => { setTimeout(() => resolve({ id: 1, token: "fake-jwt" }), 1500); }); }; // DESPUÉS (con axios real — misma firma, diferente cuerpo): export const loginUser = async (data) => { const response = await axios.post(`${API_URL}/auth/login`, data); return response.data; // { user: {...}, token: "eyJhbG..." } };
💡

¿Qué es axios y por qué no usamos fetch nativo?

fetch es nativo del navegador pero requiere más código manual: parsear JSON, manejar errores HTTP (fetch no lanza error en 401/500), configurar headers en cada llamada. axios hace todo eso automáticamente y permite crear interceptores globales para agregar el token JWT a todas las peticiones.

// Ejemplo de interceptor axios (Sprint 2): agrega el token a TODAS las peticiones axios.interceptors.request.use((config) => { const user = JSON.parse(localStorage.getItem('user')); if (user?.token) { config.headers.Authorization = `Bearer ${user.token}`; } return config; }); // Ahora TODAS las llamadas de authService incluyen el token automáticamente

📁 /utils/ — Utilidades y Validaciones

La carpeta /utils/ contiene funciones y constantes de propósito general que no pertenecen a ningún componente específico.

validators.js util · zod

Define los schemas de validación usando la librería Zod. Un schema es la descripción de qué datos son válidos para cada formulario.

  • loginSchema: valida email (formato válido) + password (mínimo 6 caracteres)
  • registerSchema: valida nombre (min 2 chars) + apellido (min 2 chars) + email + password
  • Se conecta a react-hook-form via zodResolver del paquete @hookform/resolvers
  • Los errores se muestran automáticamente debajo de cada campo del formulario
  • La validación ocurre en el cliente (antes de enviar al servidor) — ahorra peticiones innecesarias
// ¿Cómo se conecta validators.js con react-hook-form? import { useForm } from "react-hook-form"; import { zodResolver } from "@hookform/resolvers/zod"; import { loginSchema } from "../../utils/validators"; function LoginPage() { const { register, handleSubmit, formState: { errors } } = useForm({ resolver: zodResolver(loginSchema) // ← Conecta el schema }); const onSubmit = (data) => { // Solo llega aquí si TODOS los campos pasaron la validación login(data); }; return ( <form onSubmit={handleSubmit(onSubmit)}> <input {...register("email")} /> {errors.email && <p>{errors.email.message}</p>} // "Debe ser un email válido" <input type="password" {...register("password")} /> {errors.password && <p>{errors.password.message}</p>} // "Mínimo 6 caracteres" </form> ); }
